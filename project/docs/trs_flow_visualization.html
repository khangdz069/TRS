TRS Rebuild: hiểu trong 10 phút

Đây là bản tóm tắt dễ đọc để báo cáo: xem project như một phần mềm bình thường, nhưng có lõi nghiên cứu nằm ở phần gợi ý testcase.

Frontend Backend Grader Recommendation Python model service

1. Nói một câu là gì?

TRS Rebuild là hệ thống web cho sinh viên nộp code C++, hệ thống chấm bằng testcase, rồi gợi ý một vài testcase sai quan trọng để sinh viên sửa bài hiệu quả hơn.

Sinh viên

Nộp file code, xem kết quả đúng/sai, xem testcase được gợi ý và gửi feedback.

Giảng viên

Tạo assignment, import danh sách sinh viên, xem phản hồi và thống kê.

Hệ thống

Backend nhận request, gọi grader, lưu kết quả và gọi recommendation engine.

2. Luồng quan trọng nhất: sinh viên nộp bài

1
Student submit Frontend gửi bài lên backend.
2
Backend nhận Route submission kiểm tra dữ liệu.
3
Gọi grader Backend gửi code sang service chấm.
4
Compile C++ Grader biên dịch và chạy testcase.
5
Lưu kết quả Backend lưu pass/fail từng testcase.
6
Gợi ý Recommendation chọn tối đa 3 testcase sai.
7
Feedback Sinh viên đánh giá gợi ý có hữu ích không.

3. Phần mềm bình thường hay NCKH?

Phần mềm bình thường
  • Có màn hình đăng nhập, teacher, student.
  • Có backend API.
  • Có database lưu account, assignment, submission.
  • Có grader chấm code C++.
Lõi NCKH
  • Không chỉ trả về đúng/sai.
  • Hệ thống chọn testcase sai nào nên gợi ý.
  • Việc chọn dựa trên nhóm người học và matrix prediction.
  • Mục tiêu là hỗ trợ sinh viên sửa bài tốt hơn.

4. Model Python chạy thế nào?

Mỗi lần sinh viên nộp bài, hệ thống không chạy cả 3 mô hình. Nó xác định sinh viên thuộc nhóm nào rồi chọn một mô hình tương ứng.

Nhóm sinh viên Model được chọn Ý nghĩa khi báo cáo
apr1 RSVD Dùng matrix dự đoán của RSVD để xếp hạng testcase fail.
apr2 timeSVD Dùng matrix dự đoán của timeSVD cho nhóm này.
apr3 LSTM Dùng matrix dự đoán của LSTM cho nhóm này.
unknown RSVD Nếu không tìm được nhóm thì fallback về RSVD.

Backend Java giữ các rule nghiệp vụ, sau đó gọi model Python để chọn group, chọn RSVD/timeSVD/LSTM và rank testcase fail bằng matrix.

5. Recommendation chọn testcase như thế nào?

1
Lấy testcase fail Ví dụ sinh viên sai 20 testcase.
2
Kiểm tra rule Kiểm tra testcase cũ, daily limit và trạng thái submission.
3
Gọi model Backend gọi model-python qua HTTP.
4
Lấy top testcase Model trả tối đa 3 testcase fail có điểm cao.
5
Sort Sắp xếp testcase fail theo điểm giảm dần.
6
Lấy top 3 Chỉ gợi ý tối đa 3 testcase.
7
Hiển thị Student xem testcase và gửi feedback.

6. File nào ứng với phần nào?

Phần File nên đọc Đọc để hiểu gì?
Màn hình sinh viên frontend/src/app/student/page.tsx Nút nộp bài, lịch sử submission, hiển thị recommendation.
Backend nhận submission backend/src/main/java/com/trs/backend/controller/SubmissionController.java Nhận bài nộp, gọi grader, lưu kết quả.
Máy chấm grader/src/main/java/com/trs/grader/GraderService.java Compile C++, chạy testcase, trả về pass/fail.
Logic gợi ý backend/src/main/java/com/trs/backend/service/RecommendationService.java Chọn nhóm, chọn model, lấy top 3 testcase.
Load matrix model model-python/app.py Load matrix model và rank testcase fail.

7. Câu nói mẫu khi báo cáo

Project này có thể xem là một hệ thống phần mềm web bình thường ở phần giao diện, backend, database và grader. Tuy nhiên điểm nghiên cứu nằm ở recommendation engine: sau khi sinh viên nộp bài và có danh sách testcase sai, hệ thống dùng nhóm người học và model Python để chọn tối đa 3 testcase phù hợp nhất nhằm hỗ trợ sinh viên sửa bài.

Nên nói

  • Hệ thống chọn một model theo nhóm sinh viên.
  • Model Python load matrix dự đoán từ file .npz.
  • Recommendation chỉ xếp hạng các testcase đang fail.
  • Mục tiêu là gợi ý testcase có ích, không đưa hết lỗi ra cùng lúc.

Không nên nói

  • Hệ thống chạy cả 3 model mỗi lần sinh viên nộp bài.
  • Project training model nâng cao trực tiếp lúc submit.
  • Recommendation chọn testcase ngẫu nhiên.
  • Đây chỉ là phần mềm CRUD bình thường.
